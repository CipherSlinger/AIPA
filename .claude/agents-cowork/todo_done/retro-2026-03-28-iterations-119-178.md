## 核心问题：PRD 粒度过细

### 问题描述

60 轮迭代中，绝大多数迭代只实现了一个微型功能点。每个微功能都走了完整的 PM->UI->Frontend->Tester 流水线（或至少 PM->Frontend->Tester），流水线启动开销远大于实际开发量。

### 数据支撑

**Notes 系统案例**（14 轮迭代实现一个完整的笔记功能）：

| 迭代 | 功能 | 代码量估算 |
|------|------|-----------|

| 121 | 笔记-聊天集成 | ~80 行 |
| 122 | Markdown 预览 + 搜索 | ~150 行 |
| 124 | 笔记分类 | ~250 行 |
| 125 | 组件拆分重构 | 重构,无新功能 |
| 126 | 笔记导出 | ~100 行 |

| 159 | 字数统计 | ~20 行 |
| 160 | 阅读时间 | ~20 行 |

合理的聚合方案：
- PRD-1: "Notes 核心体验" (120-122, 124-127) → 1 轮迭代
- PRD-2: "Notes 打磨" (159, 160, 166, 167, 172-175) → 1 轮迭代
- 总计 2 轮迭代 vs 实际 14 轮，效率差 7 倍

**i18n 案例**（14 轮迭代做剩余字符串国际化）：

迭代 142-157 每次修复 1-3 个组件的硬编码英文字符串。每个迭代改动不超过 20 行代码。合理做法：1 个 PRD "i18n 完善扫描"，1 轮迭代全部完成。效率差 14 倍。

### 根因分析

1. **aipa-pm 缺乏粒度约束**：PRD 模板没有最小粒度要求，允许为 10 行代码变更出一份完整 PRD
2. **agent-leader 未做粒度把关**：leader 对 PM 输出未做功能聚合审查，直接将微型 PRD 推入流水线
3. **流水线无"快速通道"**：纯 CSS 调色、i18n 补全等不需要产品设计的工作仍走完整 PM->UI->Frontend->Tester 链路

### 顺畅之处
1. **构建可靠性极高**：60 轮迭代 0 次构建失败，说明 TypeScript 类型系统和 Vite 构建链稳定
2. **代码质量一致**：前端代码风格、i18n 集成模式、组件结构保持一致
3. **功能交付持续**：从自定义模板 → 笔记系统 → 剪贴板操作 → 会话管理 → 各种微改进，功能持续交付

### 卡点与摩擦
1. **PRD 粒度失控**（P0）：核心问题。单功能微型 PRD 导致流水线启动开销 >> 实际开发量
2. **回顾会完全缺失**（P0）：60 轮无回顾，导致问题累积到用户手动指出才被处理
3. **ITERATION-LOG 格式退化**（P1）：Iteration 165-178 退化为一行摘要，tester 和 leader 无法有效审查
4. **工程任务走 PM 流程**（P1）：i18n 补全、CSS 调色等不需要产品分析的工作仍走完整流水线

### 工作流改进建议（已落地）
1. **PRD 粒度约束**：每份 PRD 必须包含 2-4 个关联功能点（写入 aipa-pm.md）
2. **Leader 粒度把关**：leader 收到 PRD 后检查 In Scope 功能点数量，不足则要求合并（写入 agent-leader.md）
3. **简化流水线通道**：纯工程任务（i18n、CSS、bug fix）可跳过 PM 和 UI 阶段（写入 agent-leader.md）
4. **ITERATION-LOG 格式强制**：禁止退化为一行摘要，必须包含四个章节（写入 aipa-frontend.md）
